# docs-distill: слой синтеза

> Вырос из Sidetrack #4 кристалла [[crystal-evidence-discipline/workitem|crystal-evidence-discipline]]
> (`promoted-to-stem`). Исходный запрос — от агента `lime-analytics` через интерком; но в ходе
> разбора выяснилось, что запрос был сформулирован уже, чем проблема.
>
> **Дизайн-сессия 2026-07-14 велась в чате.** Всё существенное перенесено сюда; чат далее не
> источник истины.

## Назначение

У суиты нет **слоя синтеза**.

Фрагменты накапливаются сами: `## Decision Log` копит решения, `docs/features/` копит фичи.
Сводное представление — «как оно устроено *сейчас*», «едины ли подходы», «что сломается, если
тронуть» — **не пересобирается никогда, потому что никто не спрашивает.** Десять feature-документов,
доехавших каждый по своему адресу, не дают вывода об общности. Синтез каждый раз собирается в
голове читателя и каждый раз испаряется.

Это **тот же закон, что мы лечили в кристалле**, этажом выше:

| | Фрагменты (копятся) | Синтез (пересобирается) |
|---|---|---|
| Внутри задачи | `## Decision Log` | `## Текущая модель` ✅ (сделано) |
| **По проекту** | `docs/features/*`, находки кристаллов | **← этого нет** |

`## Текущая модель` относится к Decision Log ровно так, как архитектурный документ — к десяти
feature-докам. Одна болезнь, одно лекарство, разные этажи.

Цель работы: научить суиту (а) **спрашивать**, систематизировано ли знание; (б) владеть слоем
синтеза, не диктуя проекту, *что* именно он синтезирует; (в) сделать распад синтеза
**обнаруживаемым**, а не полагаться на память.

**Что решено (см. DL):**

- **Скилл называется `docs-distill`** — семья `docs-*` рядом с `docs-sync` (DL #7).
- **Две фазы кристалла.** Обязательство урожая рождается вместе с *работой*, а не с кристаллом.
  Кристалл фазы захвата (`prd-prep`, может навсегда остаться `dormant`) не должен ничего (DL #1).
- **Мягко, пока не названо; жёстко, как только названо.** Nudge живёт в crystal-скиллах; энфорс —
  в `docs-distill`. Nudge склоняет вызвать скилл; скилл материализует обязательство в `- [ ]`, и
  **действующий completion-guard дальше энфорсит сам** (DL #2).
- **Настоящий вопрос — систематизация, а не маршрутизация** (DL #3). Это главный разворот сессии.
- **Суита диктует отношение, а не артефакт.** Что есть синтез — решает проект; frontmatter
  объявляет, *что покрывает* и *на какой вопрос отвечает* (DL #5).
- **Связь docs-sync ↔ docs-distill двусторонняя** (DL #7).

**Что НЕ решено — открытые развилки:**

- Дистиллят — отдельная секция кристалла или это и есть `## Текущая модель` с адресами?
  (Sidetrack #1)
- Судьба триггера `docs-sync` (он уже стоит в точке финализации — конфликт) (Sidetrack #2)
- Нужен ли дефолтный путь тира или чистый auto-discovery по frontmatter (Sidetrack #3)

### #2 / 2026-07-14 / Мягко, пока не названо; жёстко, как только названо

**Source:** user
**Basis:** user-stated
**Basis-detail:** «наверное мягкие nudge должны быть частью crystal скилов, а явный push отдельным
скилом который энфорсит поведение? Может быть мягкие nudge должны склонять к запуску этого

**Context:** Как заставить дистилляцию происходить, не завалив пользователя напоминаниями и не
обязав каждый кристалл.
**Why:** Мягкое напоминание, которое ничего не может *сделать*, обесценивается за неделю. Мягкое
напоминание, которое зовёт скилл, превращающий разговор в **обязательство**, — работает. Это ровно
та механика, что уже у побегов: пока не пойман — ничего не должен; пойман — блокирует закрытие.
**Implication:** `docs-distill`, будучи вызван, пишет находки в кристалл как `- [ ]` с адресом
назначения. Дальше **энфорсит существующий completion-guard** — новых хуков и гейтов не нужно.
Кристалл не закрывается, не отдав урожай.

### #3 / 2026-07-14 / Настоящий вопрос — систематизация знания, а не маршрутизация находок

**Source:** user
**Basis:** user-stated
**Basis-detail:** «в какой-то момент - мысль-вопрос - систематизировано ли знание. Или у нас 10
документов про фичи никак не связанные между собой, из которых нельзя сделать вывод об общности,
о единости подходов».
**Context:** Ассистент проектировал маршрутизатор: находки кристалла → разложить по адресам
(features / llm / changelog / модель).
**Why:** Маршрутизация — не та проблема. Десять feature-доков могут доехать каждый по адресу — и
синтеза всё равно не будет. Отсутствует не транспорт, а **пересборка**: слой, который отвечает
«как это устроено в целом» и который надо *перестраивать*, а не дописывать.
**Implication:** `docs-distill` — владелец слоя синтеза, а не маршрутизатор. Его вопрос:
*систематизировано ли то, что выходит наружу?* Маршрутизация остаётся, но она следствие, а не суть.
**Supersedes:** — (отменяет проектное намерение ассистента, до DL-записи не дошедшее)

### Структура тира, как она сложилась сама

```
docs/analytics/
├── README.md            ← точка входа: схема потоков + «куда идти за чем»
├── systems.md           ← ЕДИНСТВЕННОЕ место, где живут ID систем
├── event-map.md         ← синтез: как устроено СЕЙЧАС (только текущее)
├── naming-convention.md ← грамматика для НОВЫХ сущностей
├── channels/linkedin.md ← реестр + протокол проверки
├── gtm/                 ← 🤖 ГЕНЕРИРУЕТСЯ: проекция внешней системы
│   ├── snapshot.json    ← отпечаток (fingerprint) состояния
│   ├── tags/*.md        ← по файлу на каждый тег (54)
│   ├── triggers/*.md    ← (33)
│   └── variables/*.md   ← (17)
├── history/             ← археология: демонтированные схемы, читать необязательно
└── references/          ← сырые артефакты (экспорты, скрины, тикеты)
```

Это **пять разных жанров**, а не один «документ модели»:

| Жанр | Свойства |
|---|---|
| **Проекция** (`gtm/`) | генерируется из снимка, руками не править, диффабельна |
| **Синтез** (`event-map`, `README`) | проза, только текущее, ссылается на проекцию |
| **Реестр** (`systems`, `channels/`) | единый источник истины для идентификаторов |
| **Конвенция** (`naming-convention`) | правила для новых сущностей |
| **История** (`history/`) | снятое с эксплуатации, opt-in чтение |

### Механизмы, до которых мы в теории не дошли

1. **Распад ловится отпечатком, а не временем.** `snapshot.json` хранит `fingerprint` контейнера,
   меняющийся при **любой** правке. Совпал — проекция актуальна; не совпал — нужен свежий экспорт.
   Наша идея «наблюдение протухает за N дней» — грубая эвристика рядом с проверяемым фактом.
   ⚠️ Но: fingerprint есть не у всякой внешней системы. Обобщать осторожно.

2. **Пере-наблюдение = перегенерация + `git diff`.** Не «вспомни сходить посмотреть», а операция,
   чей вывод — **точный список того, что изменилось во внешней системе**.

3. **Проекция — аналитический субстрат, а не документация.** Обратный индекс проекции нашёл живой
   баг (два активных GA4-тега на одних триггерах → каждый шаг воронки уезжает дважды), **которого
   в UI самого GTM не видно**. Это, возможно, сильнейший аргумент за весь тир: grep по
   спроецированному состоянию видит то, чего не показывает родная система.

4. **Битая wiki-ссылка = сигнал устаревания снимка, и это объявлено фичей.** Проверка ссылочной
   целостности превращается в детектор распада.

5. **Ритуал начала сессии** (у них — в CLAUDE.md): сверить fingerprint, спросить пользователя,
   не менялся ли контейнер. Хэндроллнутый hydrate-хук.

### Законы, которые общее аналитики — забрать целиком

| Закон | Обоснование (их формулировка) |
|---|---|
| Только текущее; история — отдельно | см. DL #6 |
| **Только абсолютные даты** в долгоживущих документах | «"2 года назад", "недавно" через год соврут молча: их никто не перечитывает» |
| **ID живут ровно в одном месте**, остальные ссылаются | «Скопированный ID протухает молча — его не перепроверяют, потому что он "уже записан"». Цена: partner ID **чужого** аккаунта нёсся 3 года незамеченным, «потому что число выглядело правдоподобно» |
| Звать по имени, не по волатильному номеру | «"Тег 320" через год не значит ничего» |
| **Новый режим отказа → новый ассерт в скрипте, а не «запомнить на будущее»** | это буквально наша же `docs/llm/soft-guidance-vs-deterministic-gates.md`, выведенная независимо |
| Читать точку входа **до** любой правки | pre-action gate, проектная сторона |

## Sidetracks

### #1. Дистиллят — отдельная секция или это `## Текущая модель` с адресами?

**Возникло в:** DL #1, #3
**Описание:** `## Текущая модель` отвечает «во что мы верим *по задаче*» — включая то, что умрёт
вместе с задачей. Дистиллят — «что переживёт». Грань тонкая, и есть риск дублирования: одно и то же
утверждение будет жить в двух секциях и разъедется. Возможно, честнее: находки копятся в
`## Текущая модель`, а `docs-distill` **вытягивает** из неё, а не кристалл **проталкивает**. Решить
до реализации.

**Status:** open

### #2. Конфликт триггеров: `docs-sync` уже стоит в моменте финализации

**Возникло в:** DL #7
**Описание:** `docs-sync` объявляет «INVOKE BEFORE COMPLETING any task that changes user-facing
behavior» — та же секунда, на которую претендует `docs-distill`. Два скилла будут молча считать, что
второго нет. Развилка (пользователь склонился к первому, но решение не финализировано):
(а) `docs-distill` — триггер, `docs-sync` — исполнитель: снять с docs-sync роль триггера, оставив
ему работу (он остаётся тем, кто умеет писать `docs/features/`, но перестаёт быть тем, кто решает,
что пора); (б) docs-distill владеет только новым тиром, остальные живут как жили — дёшево, но
решение «что куда» остаётся бесхозным, а именно его отсутствие и убило находки у lime.
⚠️ Отдельно: триггер docs-sync («перед завершением задачи») **слишком поздний** для «дистилляции на
ходу» — это аргумент против варианта (б) и против простого расширения docs-sync.

**Status:** open

### #3. Где живёт тир: фиксированный путь или чистый auto-discovery?

**Возникло в:** DL #5
**Описание:** lime выбрал `docs/analytics/` — доменное имя под проект. Фиксировать `docs/model/`
или пусть проект называет сам, а суита ищет по frontmatter `type:` (как auto-scan ищет `tasks/`)?
Склоняемся ко второму, но тогда нужен способ объявить «вот это у меня синтез», иначе nudge не будет
знать, что с чем сравнивать. Проверить, что frontmatter-скана достаточно и он не ловит лишнего.

**Status:** open

### #4. Post-action verification: «сделал» ≠ «проверил»

**Возникло в:** полевые находки, `channels/linkedin.md`
**Описание:** У них прямым текстом: «**Галочка ставится только после проверки, а не после
"сделал"**». Наш `crystal-cut` считает `- [ ]` → `[x]` закрытием — но действие, совершённое над
внешней системой, это **новое непроверенное утверждение**. Можно закрыть кристалл, выполнив всё и
не проверив ничего. Ровно это с ними и случилось: опубликовали, отметили сделанным, а наблюдение —
то самое, которое поймало бы ошибку, — не сделали. Pre-action gate без зеркала («совершив
необратимое, пойди и посмотри, что оно сделало то, что ты думал») ловит половину класса.
**Отдельная работа**: это про *обязательство* (crystal-cut), а дистилляция — про *знание*; мешать
их в одном кристалле — размыть оба. Дёшево и злее — возможно, делать раньше.

**Status:** open

### #5. Fingerprint обобщается не на всякую систему

**Возникло в:** полевые находки, механизм 1
**Описание:** У GTM есть `fingerprint`, меняющийся при любой правке, — идеальный детектор распада.
У GA4-отчёта такого нет; у произвольной внешней системы — тем более. Значит нужна **лестница**
детекторов распада: fingerprint (лучший) → диффабельный экспорт → дата наблюдения + жест обновления
(худший, эвристика). Не выдавать частный случай за общее правило.

**Status:** open

### #6. Правило провенанса не спрашивает «а можно ли» и «а нужно ли»

**Возникло в:** DL #9
**Описание:** Отгруженное сегодня правило (`crystal-grow/SKILL.md` → «`references/` — the provenance
rule») гласит: артефакт, на который опирается DL-запись, ложится в `references/` **до** написания
записи. Оно молча предполагает, что артефакт **наш** и что хранить его **надо**. Оба предположения
сегодня же и сломались. Правилу не хватает трёх оговорок:

1. **Можно ли хранить?** Чужие / конфиденциальные артефакты не обязаны жить в твоём репозитории.
   Провенанс — не право складировать чужие данные. Конфликт с конфиденциальностью правило не видит.
2. **Нужно ли хранить?** Если письменная запись **полностью восстанавливает** утверждение (структура
   каталогов, значение конфига, список), то запись *и есть* артефакт. Хранение — для того, что проза
   не реконструирует: скриншоты, экспорты, сырые ответы, «я видел X», где X не транскрибируем.
3. **Anchoring.** Вендоринг доменно-специфичного специмена в дизайн-кристалл **общего** инструмента
   тянет дизайн в этот домен. `references/` — путь наименьшего сопротивления, по которому домен
   заползает обратно после того, как его выгнали из текста.

Формулировка-кандидат: *«Артефакт хранится, когда утверждение нельзя проверить по твоей же записи —
и когда ты вправе его хранить. Провенанс — не хоардинг».* Правило живёт в `crystal-grow`, то есть
чинить его надо в [[crystal-evidence-discipline/workitem|crystal-evidence-discipline]] (сейчас
`dormant`); но обнаружено здесь и здесь же зафиксировано, чтобы не потерялось.

**Status:** open
