### ⚡ PERF: reminders — smart-throttle for learn/changelog/docs-sync/crystal-capture (vdm v2.8.0, vdm-git v2.5.1)
Closes the noise-reduction follow-up tracked alongside the v2.7.0 work. Pre-v2.8.0 the four UserPromptSubmit reminders had different per-prompt cadences: `learn-reminder.sh` default `proactive` (fired every prompt), `changelog-reminder.sh` default `conditional` (fired every prompt while tree dirty), `docs-sync-reminder.sh` default `conditional` (same — and re-ran a heavy find+grep discovery every time), `crystal-capture-reminder.sh` (new in v2.7.0) already had its own inline throttle. Cumulative cost ≈1500 chars × N turns of extra context for zero marginal value once the disciplines were internalised. **New shared helper `lib/reminder-throttle.sh`** (mirrored byte-identical to `vdm-git/lib/`) exposes three functions: `_vdm_reminder_session_id` (extracts session_id from UserPromptSubmit JSON payload via jq, falls back to "default"), `_vdm_reminder_throttle_check <key> <seconds> [sid]` (returns 0 if still in throttle window — caller should exit silently), `_vdm_reminder_throttle_touch <key> [sid]` (resets the window after a successful emit). State files at `${TMPDIR:-/tmp}/vdm-reminder-throttle/<key>-<sid>`; mtime check uses BSD `stat -f %m` with GNU `stat -c %Y` fallback so the helper works on both macOS and Linux. **All four reminders gain a `smart` mode (now default).** Smart mode = previous fire condition (e.g. tree dirty) AND throttle window elapsed (default 600s per session, override via `<section>.throttle` for learn/changelog/docs-sync or `crystal.capture-throttle` for crystal-capture). The legacy modes (`silent`, `conditional|quiet`, `proactive`) are preserved verbatim for users who want the old behavior — e.g. `learn.mode=proactive` restores per-prompt firing. Behavioral change: first prompt of a fresh session in dirty tree → reminder fires; subsequent prompts within 10 min → silent; after 10 min idle → fires again. **crystal-capture-reminder refactored** from its inline throttle code (shipped in v2.7.0) to use the new shared helpers, removing ~25 lines of duplication. No mirror change to vdm-git plugin scripts (which only ships `guard`); the lib helpers mirror because mirror is the lib invariant. Docs-sync gate sequence reordered so the throttle check short-circuits **before** the heavy find+grep discovery — the expensive work runs at most once per throttle window. Default throttle 600s (10 min) chosen empirically: short enough to catch the moment after a real work segment, long enough that the assistant doesn't see the same hint twice in a tight loop. Verified end-to-end: first call → emits; second call within throttle → silent; gates (lib-sync, skill-paths, version-bump) green.
**Ref**: plugins/vdm/lib/reminder-throttle.sh (new), plugins/vdm-git/lib/reminder-throttle.sh (mirror), plugins/vdm/scripts/learn-reminder.sh (smart default), plugins/vdm/scripts/changelog-reminder.sh (smart default), plugins/vdm/scripts/docs-sync-reminder.sh (smart default + throttle-before-discovery), plugins/vdm/scripts/crystal-capture-reminder.sh (use shared helpers), plugins/vdm/.claude-plugin/plugin.json, .claude-plugin/marketplace.json

### ✨ FEATURE: crystal — in-flight capture discipline (capture-reminder hook + work-without-capture nudge) (vdm v2.7.0)
Closes a real gap surfaced by field experience: a session worked for several hours on an active workitem (`psalm-level-2-type-safety`), made 5+ architectural decisions and 6+ sidetracks worth of observations, and never opened `workitem.md` once. At end-of-session the assistant itself diagnosed the cause — three of four UserPromptSubmit reminders existed (docs-sync, learn, changelog) but **nothing reminded the assistant to capture decisions/observations into the active crystal mid-session**. Crystal hooks fixed only the terminal-discipline (cut/done blocked without checkbox sweep); in-flight discipline had no home. This release closes that gap on four fronts. **(1) New hook `plugins/vdm/scripts/crystal-capture-reminder.sh` registered on UserPromptSubmit.** Smart by default — fires only when source files under project root are newer than the active `workitem.md` (signal: "work happening without capture"), with per-session-id throttle (default 600s, configurable via `crystal.capture-throttle`). Three modes via `crystal.capture-mode`: `smart` (default), `proactive` (every prompt), `silent`. Tokens-aware: does *not* mechanically fire every prompt — that would cost ~500 chars × N turns for zero marginal value once discipline is internalised. Excludes resolved crystal roots, `.git/`, `node_modules/`, `vendor/`, `.claude/`, `.serena/`, `.obsidian/` from the source-newer scan. Fails open on stat/find/jq absence. **(2) `crystal-stop-reminder.sh` upgrade** — same source-newer-than-workitem heuristic at end-of-turn, appends a `📌 <slug>: source edited but workitem.md untouched — decisions taken or observations worth recording?` nudge below the existing open-items list. Catches the case from the field report: long Stop fires the open-items reminder, now also asks whether the work that happened needs capture. **(3) `crystal-hydrate.sh` SessionStart header strengthened** — from "Read the workitem file before continuing" to "Read it before continuing — workitem.md = source of truth (every decision → `## Decision Log`, every observation → `/vdm:crystal-bud`; chat decays under compaction)". Makes the source-of-truth obligation explicit at session boot, not implicit. **(4) `crystal-grow/SKILL.md` gains a new "Working inside an active crystal (in-flight discipline)" H2 section** between Forced-promotion signals and Storage layout, with two explicit trigger tables (what counts as a DL entry — chose-X-over-Y, raised threshold, deviated from plan, user-confirmed non-obvious choice, single-instance exception, dropped earlier-in-scope work; what counts as a побег — adjacent code observation, ecosystem block, "we should also...", implicit dep, failed attempt with diagnosis, tooling bug worked around), the DL-entry format mirroring existing entries in `crystal-design/workitem.md`, a per-turn cadence reflex question in Russian (matching how the user prompts the agent), and a configuration table for the three modes. **(5) `crystal-bud/SKILL.md` description field strengthened** with explicit trigger phrases (ecosystem block, threshold change, dropped work, etc.) drawn from the field-report; higher proactive-invocation rank in skill selection. Out of scope (tracked as follow-up): retrofit the same smart-throttle pattern to `learn-reminder.sh` (currently default `proactive` — fires every prompt) and `changelog-reminder.sh` (default `conditional` already smarter but could use throttle). State files for capture-reminder throttle live in `${TMPDIR:-/tmp}/vdm-crystal-capture/<session_id>`; cleaned naturally by OS tmp policies. No lib/ changes — both hooks reuse existing `resolve_crystal_roots`, `find_workitems`, `filter_status`, `extract_slug` from `lib/crystal-path.sh`; no mirror update to vdm-git/lib/ required.
**Ref**: plugins/vdm/scripts/crystal-capture-reminder.sh (new), plugins/vdm/scripts/crystal-stop-reminder.sh (work-without-capture nudge), plugins/vdm/scripts/crystal-hydrate.sh (source-of-truth header), plugins/vdm/hooks/hooks.json (UserPromptSubmit registration), plugins/vdm/skills/crystal-grow/SKILL.md (Working inside an active crystal section), plugins/vdm/skills/crystal-bud/SKILL.md (description triggers), plugins/vdm/.claude-plugin/plugin.json, .claude-plugin/marketplace.json

### 📝 DOCS: crystal-grow "Migrating legacy docs" — two pitfalls when choosing the path (vdm v2.5.2)
`plugins/vdm/skills/crystal-grow/SKILL.md` "Migrating legacy docs" section gains a new "Two pitfalls when choosing the path" subsection between the three-path table (move-to-references / inline-rewrite / delete) and the existing "Two rules that bite during migration" subsection. Surfaced by field feedback from a real ObsidianVault migration session (2026-05-31, `crystal-suite-migration` workitem Decision Log #9 and #10) where the agent's defaults caused mid-migration pushback twice: (1) **Slug rename rule** — legacy filenames are often doc-type tags (`PRD.md`, `TODO.md`, `SPEC.md`, `NOTES.md`) with zero content signal; default of preserving filename as slug yields `slug: PRD` which double-violates (kebab-case lowercase convention *and* meaningless). Rule: rename to descriptive content slug based on what the work *does*, e.g. `manual-pipeline/PRD.md` → `pdf-archive-pipeline/workitem.md`. (2) **Brainstorm-as-reference pattern** — brainstorm transcripts, design-discussion transcripts, and historical decision-log artifacts often *look* like workitems by structure (may even contain a `## Decision Log`) but are frozen content with no future obligations. Workitem = unit with future actions; reference = frozen content, no own future. Default to **move-to-references** under the parent workitem, not **inline-rewrite** as a sibling — real example: `page-snapshot/brainstorm-2026-05-01.md` was wrongly promoted to a sibling workitem on first pass, later demoted to `snapshot-engine/references/`. Surgical edit: existing table and "Two rules that bite during migration" subsection preserved verbatim. Also in same patch — new optional `description: "<one-liner for cave/base overview>"` frontmatter field threaded through four touchpoints: (a) `crystal-grow/SKILL.md` Step 3 frontmatter block (positioned between `slug:` and `status:`, marked `# optional`); (b) `crystal-grow/SKILL.md` Step 4 (Seed from shadow) — derivation rules: on shadow auto-promotion, derive from the assistant's tracked session intent; on explicit grow, derive from invocation context or leave blank; ≤80 chars, verb-leaning ("what the work *does*", not "what it *is*"); (c) `templates/workitem-template.md` — adds the commented-out optional line matching the existing `# superseded-by:` precedent (uncommented and filled when meaningful); (d) `crystal-cave/SKILL.md` — new "Description rendering" rule under Overview mode: when present, overview rows append `— "<description>"` after slug; detail mode prints as sub-line under header; absent/empty stays unchanged (no placeholder). Improves cave/base overview signal-to-noise when workitems accumulate siblings.

### ✨ FEATURE: crystal-* multi-root auto-scan + 4-tier status taxonomy + full subcommand surface (vdm v2.5.0, vdm-git v2.5.0)
Crystal-* suite expands from single-root (`docs/tasks/`) to multi-root layouts (monorepo `packages/*/tasks/`, Obsidian-vault `projects/*/tasks/`) without explicit configuration: `resolve_crystal_roots` auto-scans `<project_root>` for any `tasks/` directory, pruning hidden segments (`.git/`, `.stversions/`, `.obsidian/`, `.trash/`, etc.) and common dependency dirs (`node_modules/`, `vendor/`). Two-implementation strategy in `_auto_scan_tasks_dirs`: `git ls-files | _extract_tasks_dirs_from_git_files` when inside a git work tree (~30ms on 7600-file vault, 130× faster than find), `find` fallback for non-git repos. Process-scope memoization in `resolve_crystal_roots` ensures O(1) for repeated calls within a single hook invocation. Config schema gains `crystal.paths` (array of globs, takes precedence), `crystal.status-aliases` (map for project-specific vocabulary → canonical), `crystal.singleton` (override `global`/`per-root`/`off`/`auto`). Backward-compat: legacy `crystal.path` (singular) still works; default behavior with no config is `enabled: true` + auto-scan. Slug qualification in multi-root mode: `<parent>/<slug>` (e.g. `auth/refactor-jwt`) to disambiguate same-named workitems across roots. Singleton invariant derives from #roots: 1 → global (DL #11 in crystal-design), ≥2 → per-root (per-root violation reported separately in hooks). Output formatting (DL #13 in crystal-multi-root): single-root keeps flat one-liner; multi-root emits group-by-root multi-line layout, alphabetical sorting within and across groups. Canonical 4-tier status taxonomy (DL #10): Pre-work (`idea`, `draft`, `ready` — visible in cave Backlog, no singleton), Active (`in-progress` — singleton), Paused (`blocked`, `dormant` — visible in cave Paused, mentioned in sweeps), Terminal (`done`, `cancelled`, `superseded` — hidden by default, accessible via `crystal-cave --all`). Completion gate (`crystal-completion-guard.py`) now distinguishes terminal statuses: `done` triggers unchecked-items gate (DL #4); `superseded` enforces `superseded-by: <slug>` frontmatter (new gate, DL #10); `cancelled` bypasses (explicit drop). ### 🐛 BUG: substring-match false-positives in git-guard hook (vdm-git v2.3.0)
The PreToolUse hook used `re.search(r"git\s+commit", command)` against the raw Bash command string, blocking on any substring match — including `grep "git commit" file`, comments containing `# git commit`, JSON payloads / heredoc bodies that happened to embed the literal text, and `cat /var/log/git-commit.log`. Surfaced repeatedly while developing the v2.3.0 auto-prep workflow (one early agent turn was blocked by its own `grep -rn "git commit" plugins/...`). Replaced with `_command_invokes(command, op)`: strips heredoc bodies, line comments, and quoted strings (single + double, with backslash-escape awareness in double quotes), then matches `git\s+<op>\b` only at command boundaries (start-of-string, whitespace, `;`, `&&`, `||`, `|`, `(`, backtick). Validated by `scripts/test-git-guard-hook.py` — 31 cases, mix of true-positives (chained, subshell, backtick-substitution, if-block) and false-positives (quoted args, comments, heredoc bodies, longer command names like `gitk`, `git-commit.log` paths, URLs).
