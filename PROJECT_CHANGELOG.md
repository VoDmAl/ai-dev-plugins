### тЪб PERF: reminders тАФ smart-throttle for learn/changelog/docs-sync/crystal-capture (vdm v2.8.0, vdm-git v2.5.1)
Closes the noise-reduction follow-up tracked alongside the v2.7.0 work. Pre-v2.8.0 the four UserPromptSubmit reminders had different per-prompt cadences: `learn-reminder.sh` default `proactive` (fired every prompt), `changelog-reminder.sh` default `conditional` (fired every prompt while tree dirty), `docs-sync-reminder.sh` default `conditional` (same тАФ and re-ran a heavy find+grep discovery every time), `crystal-capture-reminder.sh` (new in v2.7.0) already had its own inline throttle. No mirror change to vdm-git plugin scripts (which only ships `guard`); the lib helpers mirror because mirror is the lib invariant. Docs-sync gate sequence reordered so the throttle check short-circuits **before** the heavy find+grep discovery тАФ the expensive work runs at most once per throttle window. Default throttle 600s (10 min) chosen empirically: short enough to catch the moment after a real work segment, long enough that the assistant doesn't see the same hint twice in a tight loop. Verified end-to-end: first call тЖТ emits; second call within throttle тЖТ silent; gates (lib-sync, skill-paths, version-bump) green.

### тЬи FEATURE: crystal тАФ in-flight capture discipline (capture-reminder hook + work-without-capture nudge) (vdm v2.7.0)
Closes a real gap surfaced by field experience: a session worked for several hours on an active workitem (`psalm-level-2-type-safety`), made 5+ architectural decisions and 6+ sidetracks worth of observations, and never opened `workitem.md` once. At end-of-session the assistant itself diagnosed the cause тАФ three of four UserPromptSubmit reminders existed (docs-sync, learn, changelog) but **nothing reminded the assistant to capture decisions/observations into the active crystal mid-session**. Crystal hooks fixed only the terminal-discipline (cut/done blocked without checkbox sweep); in-flight discipline had no home. This release closes that gap on four fronts. **(1) New hook `plugins/vdm/scripts/crystal-capture-reminder.sh` registered on UserPromptSubmit.** Smart by default тАФ fires only when source files under project root are newer than the active `workitem.md` (signal: "work happening without capture"), with per-session-id throttle (default 600s, configurable via `crystal.capture-throttle`). Three modes via `crystal.capture-mode`: `smart` (default), `proactive` (every prompt), `silent`. Tokens-aware: does *not* mechanically fire every prompt тАФ that would cost ~500 chars ├Ч N turns for zero marginal value once discipline is internalised. Excludes resolved crystal roots, `.git/`, `node_modules/`, `vendor/`, `.claude/`, `.serena/`, `.obsidian/` from the source-newer scan. Fails open on stat/find/jq absence. **(2) `crystal-stop-reminder.sh` upgrade** тАФ same source-newer-than-workitem heuristic at end-of-turn, appends a `ЁЯУМ <slug>: source edited but workitem.md untouched тАФ decisions taken or observations worth recording?` nudge below the existing open-items list. Catches the case from the field report: long Stop fires the open-items reminder, now also asks whether the work that happened needs capture. **(3) `crystal-hydrate.sh` SessionStart header strengthened** тАФ from "Read the workitem file before continuing" to "Read it before continuing тАФ workitem.md = source of truth (every decision тЖТ `## Decision Log`, every observation тЖТ `/vdm:crystal-bud`; chat decays under compaction)". Makes the source-of-truth obligation explicit at session boot, not implicit. **(4) `crystal-grow/SKILL.md` gains a new "Working inside an active crystal (in-flight discipline)" H2 section** between Forced-promotion signals and Storage layout, with two explicit trigger tables (what counts as a DL entry тАФ chose-X-over-Y, raised threshold, deviated from plan, user-confirmed non-obvious choice, single-instance exception, dropped earlier-in-scope work; what counts as a ╨┐╨╛╨▒╨╡╨│ тАФ adjacent code observation, ecosystem block, "we should also...", implicit dep, failed attempt with diagnosis, tooling bug worked around), the DL-entry format mirroring existing entries in `crystal-design/workitem.md`, a per-turn cadence reflex question in Russian (matching how the user prompts the agent), and a configuration table for the three modes. **(5) `crystal-bud/SKILL.md` description field strengthened** with explicit trigger phrases (ecosystem block, threshold change, dropped work, etc.) drawn from the field-report; higher proactive-invocation rank in skill selection. Out of scope (tracked as follow-up): retrofit the same smart-throttle pattern to `learn-reminder.sh` (currently default `proactive` тАФ fires every prompt) and `changelog-reminder.sh` (default `conditional` already smarter but could use throttle). State files for capture-reminder throttle live in `${TMPDIR:-/tmp}/vdm-crystal-capture/<session_id>`; cleaned naturally by OS tmp policies. No lib/ changes тАФ both hooks reuse existing `resolve_crystal_roots`, `find_workitems`, `filter_status`, `extract_slug` from `lib/crystal-path.sh`; no mirror update to vdm-git/lib/ required.

### ЁЯУЭ DOCS: crystal-grow "Migrating legacy docs" тАФ two pitfalls when choosing the path (vdm v2.5.2)
`plugins/vdm/skills/crystal-grow/SKILL.md` "Migrating legacy docs" section gains a new "Two pitfalls when choosing the path" subsection between the three-path table (move-to-references / inline-rewrite / delete) and the existing "Two rules that bite during migration" subsection. Surfaced by field feedback from a real ObsidianVault migration session (2026-05-31, `crystal-suite-migration` workitem Decision Log #9 and #10) where the agent's defaults caused mid-migration pushback twice: (1) **Slug rename rule** тАФ legacy filenames are often doc-type tags (`PRD.md`, `TODO.md`, `SPEC.md`, `NOTES.md`) with zero content signal; default of preserving filename as slug yields `slug: PRD` which double-violates (kebab-case lowercase convention *and* meaningless). Rule: rename to descriptive content slug based on what the work *does*, e.g. `manual-pipeline/PRD.md` тЖТ `pdf-archive-pipeline/workitem.md`. (2) **Brainstorm-as-reference pattern** тАФ brainstorm transcripts, design-discussion transcripts, and historical decision-log artifacts often *look* like workitems by structure (may even contain a `## Decision Log`) but are frozen content with no future obligations. Workitem = unit with future actions; reference = frozen content, no own future. Default to **move-to-references** under the parent workitem, not **inline-rewrite** as a sibling тАФ real example: `page-snapshot/brainstorm-2026-05-01.md` was wrongly promoted to a sibling workitem on first pass, later demoted to `snapshot-engine/references/`. Surgical edit: existing table and "Two rules that bite during migration" subsection preserved verbatim. ### тЬи FEATURE: crystal-* multi-root auto-scan + 4-tier status taxonomy + full subcommand surface (vdm v2.5.0, vdm-git v2.5.0)
Crystal-* suite expands from single-root (`docs/tasks/`) to multi-root layouts (monorepo `packages/*/tasks/`, Obsidian-vault `projects/*/tasks/`) without explicit configuration: `resolve_crystal_roots` auto-scans `<project_root>` for any `tasks/` directory, pruning hidden segments (`.git/`, `.stversions/`, `.obsidian/`, `.trash/`, etc.) and common dependency dirs (`node_modules/`, `vendor/`). Two-implementation strategy in `_auto_scan_tasks_dirs`: `git ls-files | _extract_tasks_dirs_from_git_files` when inside a git work tree (~30ms on 7600-file vault, 130├Ч faster than find), `find` fallback for non-git repos. Process-scope memoization in `resolve_crystal_roots` ensures O(1) for repeated calls within a single hook invocation. Config schema gains `crystal.paths` (array of globs, takes precedence), `crystal.status-aliases` (map for project-specific vocabulary тЖТ canonical), `crystal.singleton` (override `global`/`per-root`/`off`/`auto`). Backward-compat: legacy `crystal.path` (singular) still works; default behavior with no config is `enabled: true` + auto-scan. Slug qualification in multi-root mode: `<parent>/<slug>` (e.g. `auth/refactor-jwt`) to disambiguate same-named workitems across roots.
